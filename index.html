SAFEPOS · KTP Objective

The Problem Space · As I See It

01
Sensor Deception, Not Sensor Failure
In a metal-dense port, the dangerous failure mode isn't a sensor going offline.
What happens when the sensor is confidently wrong and the system has no reason to doubt it?
02
Weather as an Operational Gate
Port schedules don't pause for marginal conditions.
Who decides when it's safe to fly, on what evidence, and how is that decision defensible to the CAA?
03
Coordination Fragility
Independent policies only stay safe if each agent has accurate state information.
If one drone lies about its position mid-lift, what does that do to every drone coordinating against it?
04
Operational Authority at Scale
A single BVLOS mission has a clear authority chain. A twelve-drone fleet mid-operation over an active dock does not.
When a crane swings into a flight path, who overrides — the operator, the system, or neither?
These aren't edge cases. They are the design constraints.

08 / 11
Architecture Decisions

Teaching Drones to
Coordinate Without a
Central Controller

Each drone: own state + neighbour observations → independent policy → residual thrust + moment output
Policy Type
Independent per agent · no central controller. Centralised control is a single point of failure. Doesn't scale to a port fleet.
State Space
Own state + neighbour observations. Coordination emerges from training · not hard-coded rules that break on edge cases.
Action Space
Residual thrust + moment over a structured baseline. Better training stability. Maps cleanly to real actuator constraints.
Training Condition
Crate mass and size randomised throughout training. Variable payloads are the training condition · not an edge case the policy sees once.
